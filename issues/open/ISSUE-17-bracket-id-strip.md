# ISSUE-17 角括弧の構造ラベル・IDが除外されず誤検出される

- **深刻度**: 高（実運用中の物理論文で大量の誤検出を生む）
- **分類**: 抽出方針の不足（実運用で発覚）
- **branch**: fix/17-bracket-id-strip

## 症状・実害
実際の論文（「観測の窓」v2、`v2/source/ch01〜ch09.md`）に explain-lint を適用したところ、
以下のような角括弧タグが LATIN 抽出に引っかかり、大量の偽用語を生成した（実測）:

- `[PREM]`（201箇所）、`[CLOS]`（174）、`[CITE]`（118）、`[OGEO]`（54）、`[XVER]`（45）、
  `[OPHY]`（29）、`[OIMP]`（26）、`[HYPO]`（24）、`[CORE]`（12）など閉包分類ラベル
- `[LEM:S4-FIVEELEVEN]`、`[THM:STEINWEISS]`、`[FACT:S5-IRREP7]`、`[OBS:WINDOW-M-OPEN]` 等の
  名前付きID（コロン以降が `FIVEELEVEN`／`STEINWEISS`／`DLAP-HIER` のような断片語として抽出される）

これらは読者向けの説明が必要な「用語」ではなく、文書の構造マーカー（論証ラベル・補題ID）。
台帳が構造ノイズで埋まり、本来判断すべき本文語がの判断コストに埋没する。

旧 `check_terms.py`（このツールのスピンオフ元、論文プロジェクト内の独立実装）には
`[(?:DEF|LEM|THM|OBS|FACT):[A-Z0-9-]+]` という除外パターンがあったが、
これは既知の5種類のプレフィックスのみのハードコーディングであり、かつ explain-lint 側には
そもそも移植されていなかった。

## 根本原因
`patterns.py` の `STRIP` リストは数式・コード・画像・URL は除外するが、
「角括弧で囲まれた全大文字トークン（+ 任意でコロン区切りの識別子）」という
构造マーカー全般を汎用的に除外する仕組みがない。

## 修正方針
- `STRIP` に汎用パターンを追加: `\[[A-Z][A-Z0-9]*(?::[A-Z0-9][A-Za-z0-9_-]*)?\]`
  （例: `[PREM]`, `[LEM:S4-FIVEELEVEN]` にマッチ。プレフィックスをハードコードしない——
  DEF/LEM/THM/OBS/FACT に限定せず、全大文字で始まる角括弧トークンを一般に構造ラベルとみなす）
- 通常の Markdown リンク `[Some Text](url)` は大文字だけの語には通常ならないため誤爆しない
  （小文字・スペースを含む語は正規表現にマッチしない）。この前提をテストで明示する。
- 既存の `AT&T`/`R&D` 等の角括弧を使わない略語抽出（ISSUE-04）には影響しない。

## 完了条件（回帰テスト）
- `test_extract`: `[PREM]`, `[CLOS]`, `[LEM:S4-FIVEELEVEN]`, `[THM:STEINWEISS]` 等が
  用語として抽出されないことを確認。
- 通常の大文字語（`NASA`, `AT&T` 等、角括弧なし）は従来通り抽出されることを確認（回帰なし）。
- 角括弧つきの通常リンクテキスト `[Some Term](url)` は本文語として引き続き拾えることを確認。
